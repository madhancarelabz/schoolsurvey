STEP 3: VPS — EMPLOYEE ROLE VALIDATION               ✅ DONE
═══════════════════════════════════════════════════════════════

ORIGINAL PLAN: n8n workflow changes (lookup EMPLOYEES tab mid-flow)
CORRECTED TO: Postgres employees table + otp.js lookup

REASON FOR CHANGE:
  n8n caches node outputs. If we set role='PENDING' and updated
  DB mid-flow, the cached value in "Switch — User Role Router"
  and "Get row(s) in sheet" would still read 'PENDING' — causing
  the workflow to BREAK. The Postgres approach sets the correct
  role from session creation, so n8n reads the correct role
  immediately. ZERO n8n changes needed for role lookup.

WHAT WE DID:
  3A. Created employees table in Postgres:
      CREATE TABLE employees (employee_id, name, role, phone)
      Inserted 3 mock records (EMP001, ADM001, SUP001)

  3B. Updated otp.js:
      • Added employee lookup: queries employees table
      • If employee_id not found → returns 404 error
      • If found → uses correct role from table
      • Removed hardcoded 'TEACHER'
      • Committed and pushed to GitHub
      • Pulled on VPS, rebuilt Docker container

IMPACT: Session created with correct role. No n8n changes needed.
VERIFIED: All 4 Docker containers running.

STEP 5: DEBUGGED n8n WORKFLOW ERRORS                  ✅ DONE
═══════════════════════════════════════════════════════════════

ISSUE 1: UUID parsing error — "invalid input syntax for type uuid"
  ROOT CAUSE: session_id was arriving as '["uuid"]' (array-wrapped
              string) instead of 'uuid'. The Code: Parse Tool Args
              node was outputting the session_id with brackets.
  FIX: Updated 5 n8n nodes to correctly extract session_id:
       • Postgres: Fetch Session
       • Postgres: Update Status
       • Postgres: Save Response
       • Postgres: Compute Final Scores
       • Postgres: Burn Session

ISSUE 2: "there is no parameter $5" in Save Response
  ROOT CAUSE: n8n comma-separated queryReplacement dropped undefined
              values (category was undefined for INIT). This caused
              fewer parameters than expected.
  FIX: 
       • Added '|| Unknown' fallback to category parameter
       • Added WHERE $2 != 'INIT' guard in SQL query
       • Updated queryReplacement expression for correct parameter mapping

IMPACT: All n8n executions now succeed. Zero errors.

═══════════════════════════════════════════════════════════════
STEP 6: DEBUGGED VAPI ASSISTANT ACCURACY              ✅ DONE
═══════════════════════════════════════════════════════════════

ISSUE: AI assistant was skipping questions, reading options from
       memory, hallucinating responses, and speaking while tool
       was processing.

ROOT CAUSE: Vapi tool config had "blocking: false" on request-start
            message. This allowed the AI to speak WHILE waiting for
            the tool result from n8n, causing it to predict/hallucinate
            instead of reading the actual question text.

FIXES APPLIED:

  A. Vapi System Prompt:
     • Added CRITICAL WAIT RULE (Step 5): "STOP COMPLETELY after
       calling next_question. WAIT SILENTLY for tool result."
     • Strengthened rule: "NEVER speak while a tool call is in
       progress. Remain COMPLETELY SILENT."
     • Added rule: "NEVER read options from memory."
     • Removed "immediately" from Smart Confirmation step 4a
     • Changed option format from "ഒന്ന്. [option]" to 
       "ഒന്ന്: [option]" for stronger number prefix
     • Added: "ALL FOUR options MUST be read COMPLETELY every time.
       NEVER stop after reading only 2 or 3 options."

  B. Vapi Tool Config (via Code view in dashboard):
     • next_question tool: changed blocking from false → true
     • complete_session tool: changed blocking from false → true

  C. Vapi Model Change:
     • Changed from GPT 4.1 → GPT 4o Cluster
     • GPT 4.1 was aggressively "optimizing" repetitive content
       (skipping "ഒന്ന്." prefix, cutting options short after 5-6
       questions). GPT 4o is more faithful to instructions.
     • Temperature set to 0.2 for consistency

IMPACT: All 12 questions now read correctly with full options.
